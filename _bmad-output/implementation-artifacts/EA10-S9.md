I want an end-to-end integration test that runs `OrchestratorService` on at least one fixture story from the EA6 cobaye project (create-story → dev-story → code-review) without human intervention, using the EA6 acceptance test harness as runner,

6. **AC6 — Uses EA6 acceptance harness**
   - Invoked via the EA6 acceptance test harness (`cop1 acceptance-test` or equivalent)
   - Produces the same scoring matrix metrics the EA6 harness expects (pass/fail per gate)

7. **AC7 — CI-runnable**

   - Does NOT hit real LLM APIs — uses mocked `BMADSessionPort` / recorded fixtures

## Dev Notes

### Why EA6 cobaye, not cop1-on-cop1

Per epics.md §EA10 design: cop1-on-cop1 during bring-up is fragile (circular dep, shared state). Use the EA6 cobaye project for reliable fixtures. cop1-on-cop1 happens later (Sprint 14 EA8 Distribution).

### Plan B rationale

EA6 (acceptance harness) runs in parallel with EA11 in Sprint 12. Sprint 13 (EA10) could start before EA6 finishes. Rather than gate EA10-S9 on EA6 completion, we allow Plan B (local minimal fixture + `InMemorySessionAdapter`) so EA10 can close V1-light DoD independently. Migration to EA6 fixture is a small follow-up.

### Mocked `BMADSessionPort`

Real SDK calls in CI would be slow and non-deterministic. Use `InMemorySessionAdapter` (EA9-S1) with scripted turn sequences that mimic a successful BMAD run. The point of this test is orchestrator wiring, NOT BMAD agent quality.

### Gate for EA11-S1/S2 deletion

This test passing is the **prerequisite** for deleting the legacy agent classes (EA11-S1) and removing the `workflow.useBMAD=false` path (EA11-S2). Once green and stable for a release, those deprecations can become deletions.
