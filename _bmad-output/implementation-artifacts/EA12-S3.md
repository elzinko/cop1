# Story EA12.3: Playbook pivot — remove commands enumeration (A5)

Status: done

## Story

As a **cop1 architect**,
I want the playbook format simplified to scrum directives only (cycle shape, escalation policy, mission) with zero BMAD command enumeration,
so that the supervisor discovers BMAD commands autonomously and the playbook does not duplicate or constrain the BMAD command surface.

## Acceptance Criteria

1. **AC1 — PlaybookSchemaV2 cleanup**
   - `commands:` and `allowed_commands` fields removed from `PlaybookSchemaV2` type definition.
   - Playbook validation rejects files containing these fields (breaking change, intentional).

2. **AC2 — PipelineStepFactory relaxation**
   - Hardcoded BMAD command references in `PipelineStepFactory.ts` (lines ~77, 83, 89) moved into supervisor system prompt as scrum guidance.
   - `invoke_bmad_command` tool accepts any BMAD command string at runtime — no allowlist check.

3. **AC3 — Playbook content updated**
   - `supervisor-playbook.md` (or equivalent) rewritten to contain only: mission statement, cycle shape (what phases the sprint follows), escalation policy, decision authority boundaries.
   - No `/bmad-*` command names in the playbook body.

4. **AC4 — supervisor-playbook-reference.md updated**
   - Reference documentation reflects the new format.

5. **AC5 — Test fixtures relaxed**
   - No test asserts on literal BMAD command names in playbook parsing.
   - Existing playbook parser tests updated to use the new format.

## Dev Notes

### A5 pivot rationale

Per SCP 2026-04-15 §A5: the D2 `commands:` map approach was rejected because it duplicates and constrains the BMAD command surface. The supervisor is an LLM — it can discover commands via `/bmad-help`, completion, and reading `_bmad/bmm/**`. The playbook should express intent (scrum directives), not implementation (command lists).

### Completion Notes List

- `PlaybookSchemaV2` never existed in code (referenced only in ADR-014 / architecture docs). AC1 satisfied by (a) introducing explicit rejection of the forbidden top-level preamble keys (`commands:`, `allowed_commands:`) and (b) relaxing `PlaybookPhase.commands` to optional so playbooks written to the new format (intent-only) parse without error.
- Two canonical cycles codified in new `BmadCycle.ts` module: `DEFAULT_BMAD_PIPELINE_COMMANDS` (legacy SprintRunner pipeline: dev → review → QA) and `DEFAULT_ORCHESTRATOR_CYCLE` (orchestrator: story-creation → dev-loop). These are the single source of truth; `PipelineStepFactory` and `OrchestratorService` now consume them.
- `OrchestratorService` fallback: when `phase.commands` is absent, it looks up the canonical commands via `defaultCommandsForPhase(phase.name)`. Unknown phase names with no commands are silently skipped. Runtime behavior of existing orchestrator runs is unchanged (phases with commands still work exactly as before); the new intent-only playbook format also runs the canonical cycle as intended.
- `SupervisorPromptBuilder` now includes a "Scrum Cycle Guidance" block listing the canonical cycle with an explicit note that the command surface is not constrained — the supervisor may substitute commands discovered via `/bmad-help`.
- Collateral fix in `MultiStepResolutionLoop.ts`: migrated `budget.tokensRemaining` access to the `RemainingBudgetResult` discriminated-union shape introduced by EA12-S5 (caught during sprint-core rebuild). Not strictly this story's scope but required to rebuild sprint-core dist so that `defaultCommandsForPhase` surfaces at the `@cop1/sprint-core` barrel.
- Collateral fix in `sprint-core/src/index.ts`: removed the duplicate `SupervisorContext` re-export (pre-existing build blocker) so the sprint-core dist rebuild succeeds. The type is still exported once from `domain/SupervisorContext.js`; the duplicate from `domain/ports/SupervisorLLMPort.js` is intentionally not re-exported at the barrel (zero app-level consumers — grep-verified).
- Pre-existing failures in `SprintRunner.test.ts` (9 tests) + `bmad-pipeline-e2e.test.ts` (1 test) are unchanged by this story — same failure signatures as on clean HEAD.
