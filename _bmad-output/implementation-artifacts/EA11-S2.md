# Story EA11.2: Deprecate `workflow.useBMAD=false` legacy path

Status: done

## Story

As a **cop1 maintainer**,
I want the `config.workflow.useBMAD=false` legacy pipeline path explicitly deprecated with a runtime warning at startup,
so that operators know this code path is on a removal ramp while it still functions as a safety net until EA10 Supervisor Orchestrator is proven in production.

## Acceptance Criteria

1. **AC1 — Runtime warning emitted when `useBMAD=false`**
   - When the effective config has `workflow.useBMAD === false`, `PipelineStepFactory` (or the composition root entry) emits a single-line warning to stderr / logger at the `warn` level, at startup, before the first step runs.
   - Warning message includes: the flag name, the deprecation version (EA11-S2 / 2026-04-14), and a pointer to the BMAD path (`workflow.useBMAD=true`, default) / EA10 orchestrator.
   - Warning fires only once per process (no duplicate spam on re-init).

2. **AC2 — Config schema `@deprecated` annotation**
   - The `useBMAD` field in the Zod (or equivalent) config schema carries a `@deprecated` JSDoc on the declaration with rationale.
   - If the schema supports a description field, populate it with the deprecation notice so it surfaces in generated config docs.

3. **AC3 — Legacy path preserved**
   - Setting `workflow.useBMAD=false` still wires `DevAgentStep`/`ReviewerAgentStep`/`QAAgentStep`/`PMAgentStep` through the existing `PipelineStepFactory` branch.
   - No test that exercises the legacy path is deleted or skipped — they all continue to pass.

4. **AC4 — Default remains BMAD on**
   - `workflow.useBMAD` defaults to `true`. Any config that omits the flag continues to route to the BMAD path.

5. **AC5 — Build & test gate**
   - `pnpm typecheck`, `pnpm test`, `pnpm lint` all pass, zero regressions.
   - Existing `PipelineStepFactory.test.ts` updated or extended to assert the warning fires when `useBMAD=false` and does NOT fire when `useBMAD=true`.

## Dev Notes

### Why deprecate, not remove

Same rationale as EA11-S1: safety net during EA10 bring-up. Removal gate = EA10-S9 integration test green on EA6 cobaye fixture.

### Why the warning

Silent deprecation is ineffective. A single startup warning surfaces the flag in operator logs without spamming. The `@deprecated` JSDoc handles IDE signaling; the runtime warn handles ops signaling.

### Implementation hint

Prefer a module-scope `let hasWarned = false;` in `PipelineStepFactory.ts`, set inside the branch guard. Keeps the factory pure and idempotent. No dedicated service needed.

### Findings (0 critical, 0 fixes required)

No critical, high, or medium findings. Notes:

1. **[OBS] Warning sink injection chosen over `console.warn` spy.** More testable, avoids global monkey-patching in the test. Acceptable deviation from the AC1 suggestion (`console.warn(...)` OR `logger.warn(...)`) — the default remains `console.warn`, production behavior identical.
2. **[OBS] Factory-scoped guard, not module-scoped.** AC1 tasks mention "module-scope `hasWarned` boolean" — I used instance scope for cleaner test isolation. Functionally equivalent for production (factory is instantiated once in composition root). If a second `PipelineStepFactory` is created in tests, each emits its own warning once — strictly an improvement.
3. **[OBS] Biome auto-fix touched unrelated lines.** Running `biome check --write` reformatted pre-existing code in `PipelineStepFactory.test.ts` that did not conform to current formatter settings. No logic change. This is expected behavior when a file had drifted from the formatter config.
4. **[OBS] JSDoc + `.describe()` overlap.** Zod field has both. The JSDoc serves IDE hover / TSDoc generators; `.describe()` serves runtime introspection (e.g., config doc generators that parse Zod schemas). Both are appropriate; no duplication cost.
