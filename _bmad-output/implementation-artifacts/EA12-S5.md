# Story EA12.5: Runtime hardening — budget simplification + reentrance guard (A1+A4)

Status: done

## Story

As a **cop1 orchestrator runtime**,
I want the budget model simplified to a single `max_tokens_per_night` cap and the reentrance guard to return structured errors instead of throwing,
so that the runtime is simpler to configure and errors are recoverable by the supervisor.

## Acceptance Criteria

1. **AC1 — Single budget cap**
   - Remove the 3-cap structure (`max_turns_per_workflow`, `max_tokens_per_session`, `max_duration_per_workflow_seconds`).
   - Replace with `max_tokens_per_night` (default 2,000,000). Rely on native SDK compaction for session-level management.

2. **AC2 — remaining_budget fail-fast**
   - `remaining_budget` tool in `toolCatalog.ts` must fail-fast if `budgetProvider` is absent — no `Number.POSITIVE_INFINITY` fallback.
   - Returns structured error `{ error: 'no_budget_provider' }`.

3. **AC3 — Reentrance guard configurable**
   - `cap=3` default, configurable via `budgets.max_reentrance_depth` in playbook frontmatter.
   - Playbook schema updated to accept this optional field.

4. **AC4 — Structured reentrance error**
   - Replace `throw new Error(...)` at `toolCatalog.ts:~81` with structured return: `{ error: 'reentrance_cap', escalation_required: true, depth, max }`.
   - Supervisor can read the error and escalate gracefully.

5. **AC5 — Track 3 event**
   - New event type `reentrance.cap_hit` added to the Track 3 metrics catalogue.
   - Emitted when reentrance cap is reached.

6. **AC6 — Tests**
   - Unit tests for budget fail-fast, reentrance structured error, configurable depth.

### Completion Notes List

- The 3-cap config structure (`max_turns_per_workflow`, `max_tokens_per_session`, `max_duration_per_workflow_seconds`) was documented in ADR-014 but **never implemented in code**. AC1 "Remove 3-cap" is therefore vacuous at the code level — AC1 satisfied by adding the canonical `max_tokens_per_night` surface to the playbook schema.
- `InvokeBmadCommandResult` and `RemainingBudgetResult` discriminated unions introduced. Breaking change at the type level: callers migrated in the test file; no other consumers of these tools exist in `packages/app` yet (checked via grep on `buildSupervisorMcpServer` / `buildSupervisorToolHandlers`).
- Playbook parser: simplest possible extension — added `extractNumber` helper and `extractBudgets` method that reads `max_tokens_per_night:` and `max_reentrance_depth:` lines from the preamble. Numeric underscores supported (`2_000_000`). No YAML frontmatter introduced to keep parser footprint small.
- Wiring of `budgets.max_reentrance_depth` into `deps.maxReentrance` at composition time is the orchestrator's responsibility — not this story's (the orchestrator composition is not yet built; existing code already accepts `deps.maxReentrance`).
- Pre-existing test failures in `packages/app/src/composition/__tests__/SprintRunner.test.ts` (9 tests) and `packages/app/src/integration-tests/bmad-pipeline-e2e.test.ts` (1 test) are unchanged by this story — verified by `git stash` + test run on clean HEAD. Documented in session log.
