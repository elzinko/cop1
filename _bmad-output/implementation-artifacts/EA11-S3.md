# Story EA11.3: Extract technical services (WorktreeService, HistoryService, StepByStepController)

Status: done

## Story

As a **cop1 architect preparing EA10 Supervisor Orchestrator**,
I want the technical concerns currently scattered inside `SprintRunner` and `DevAgent` (worktree lifecycle, session/history I/O, step-by-step control) extracted into dedicated services with clean interfaces,
so that the LLM supervisor and the orchestrator can invoke these capabilities via the tool mechanism decided in ADR-014 without depending on agent-centric implementations, and so that unit testing them in isolation becomes trivial.

## Acceptance Criteria

1. **AC1 — `WorktreeService` extracted**
   - New class `WorktreeService` in `packages/sprint-core/src/features/dev-agent/application/WorktreeService.ts` (co-located with the existing `WorktreeManager` infrastructure)
   - Encapsulates current logic from `WorktreeManager` (create, cleanup, path resolution) behind a clean interface (port in `packages/sprint-core/src/features/dev-agent/domain/ports/WorktreePort.ts`; `WorktreeManager` becomes the primary adapter)
   - `SprintRunner` and `DevAgent` (legacy) keep using the same public behavior — internally they delegate to `WorktreeService`
   - Unit tests with a filesystem fixture cover create / cleanup / re-entry

2. **AC2 — `HistoryService` extracted**
   - New class `HistoryService` in `packages/sprint-core/src/features/bmad-orchestration/application/HistoryService.ts` (co-located with existing `SessionLogger` / `SessionHistoryReader`)
   - Wraps the existing `SessionLogger` / `SessionHistoryReader` read/write surface currently inlined in callers
   - Exposes: `recordExchange(sessionId, interaction)`, `readHistory(sessionId)`, `listSessions()`
   - Integrates with existing `EventBus` (emits `history.exchange.recorded` event)
   - Unit tests with in-memory adapter

3. **AC3 — `StepByStepController` extracted**
   - New class `StepByStepController` in `packages/sprint-core/src/features/workflow/application/StepByStepController.ts`
   - Provides `awaitApproval(context)` / `setMode(mode)` / `isPaused()` primitives used by intra-command step-by-step (EA8-S5 pattern) and prepared for inter-command use (EA10-S5)
   - Mode enum: `none` | `intra` | `inter` | `both`
   - Unit tests cover mode transitions and approval flow

4. **AC4 — `CheckpointService` unchanged**
   - `CheckpointService` remains as-is. Do NOT refactor.
   - Referenced in dev notes for consistency — the 4 services form the technical-services surface that ADR-014 will expose.

5. **AC5 — Existing consumers migrated incrementally**
   - `SprintRunner` in `packages/app/src/composition/SprintRunner.ts` now takes `WorktreeService`, `HistoryService`, `StepByStepController` via constructor injection
   - Legacy `DevAgent` receives a `WorktreeService` via constructor (keeps backwards-compat default: instantiates a default one if not injected)
   - Composition root wires the new services

6. **AC6 — Public API preserved**
   - All existing public methods on `SprintRunner` / `DevAgent` keep their signature.
   - All existing tests pass without modification OR are adjusted to inject the new services as mocks.

7. **AC7 — Build, test, lint gate**
   - `pnpm typecheck` / `pnpm test` / `pnpm lint` all pass, zero regressions on the 651+ existing tests.

## Dev Notes

### Scope boundary

"Extract" not "redesign." Do NOT change behavior. This story is strictly a refactor to prepare for ADR-014 tool exposure. Behavior-level improvements belong to later stories.

### Why keep `CheckpointService` as-is

It already has a clean shape. Modifying it adds risk without benefit. It will ride on the same tool-exposure mechanism as the 3 new services (decided in EA11-S5).

### Feature placement — co-locate, do not move (Option A)

Rather than creating new `features/worktree/`, `features/history/`, `features/step-by-step/` feature modules (which would trigger import churn across the 19+ useBMAD-mentioning files and beyond), co-locate each new service next to its existing infrastructure:

```
packages/sprint-core/src/features/dev-agent/
├── application/WorktreeService.ts        ← NEW
├── domain/ports/WorktreePort.ts          ← NEW
└── infrastructure/WorktreeManager.ts     ← existing, becomes the primary adapter

packages/sprint-core/src/features/bmad-orchestration/
├── application/HistoryService.ts         ← NEW
├── application/SessionLogger.ts          ← existing, consumed by HistoryService
└── application/SessionHistoryReader.ts   ← existing, consumed by HistoryService

packages/sprint-core/src/features/workflow/
└── application/StepByStepController.ts   ← NEW
```

This preserves existing imports and keeps blast radius minimal. If a dedicated feature boundary becomes warranted later (V1.1+), the services are already cleanly separated in files and can be relocated in one rename pass.

### Migration approach

Side-by-side introduction. Do not delete old paths until the new services are wired and tested. `DevAgent`'s default `WorktreeManager` instantiation (constructor default) stays functional — the service is injected in `SprintRunner` to give the orchestrator a clean hook.
