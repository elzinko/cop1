# Story EA11.6: SupervisorContext bootstrap loader

Status: done

## Story

As a **cop1 supervisor starting a sprint**,
I want a bootstrap loader that reads the PRD, the architecture document, and project metadata from the filesystem and injects them into supervisor sessions via a `SupervisorContext` structure,
so that the LLM supervisor makes decisions with real project grounding from turn 1, not a hand-stitched prompt, and so that EA10 stories consume a ready-made context object rather than each re-discovering these artifacts.

## Acceptance Criteria

1. **AC1 — `SupervisorContext` type exists**
   - `SupervisorContext` type/interface in `packages/sprint-core/src/features/bmad-orchestration/domain/SupervisorContext.ts` (co-located with `SupervisorService`)
   - Fields: `{ prd: string, architecture: string, projectMetadata: { name, version, packageManager, rootPath }, iamthelaw: string (reserved, empty until EA7 follow-up), loadedAt: ISO-8601 string }`

2. **AC2 — `SupervisorContextLoader` loads from filesystem**
   - Class `SupervisorContextLoader` in `packages/sprint-core/src/features/bmad-orchestration/application/SupervisorContextLoader.ts` (next to `SupervisorService`)
   - `load(projectRoot: string): Promise<SupervisorContext>`
   - Reads:
     - PRD from `_bmad-output/planning-artifacts/prd.md` (resolve via `{planning_artifacts}` from `_bmad/bmm/config.yaml`)
     - Architecture from `_bmad-output/planning-artifacts/architecture.md`
     - Project metadata from root `package.json` + `cop1.config.yaml`
   - If a file is missing: the field is an empty string AND a warning is logged (not an error). Loader does not throw.

3. **AC3 — Supervisor session injection hook**
   - The supervisor session startup (where `BMADSessionPort.startSession` is called for supervisor-tagged sessions) accepts a `SupervisorContext` and prepends a structured system context block to the initial prompt.
   - Context block format: a fenced code block `<supervisor-context>…</supervisor-context>` with the four fields, labeled clearly.

4. **AC4 — `iamthelaw` reserved, empty**
   - Field is present in `SupervisorContext` but populated with an empty string for V1-light.
   - TODO comment referencing EA7 follow-up.

5. **AC5 — Unit tests with fixture filesystem**
   - Test file `packages/sprint-core/src/features/bmad-orchestration/__tests__/SupervisorContextLoader.test.ts`
   - Uses a temp directory with fake PRD/architecture/package.json fixtures
   - ≥ 5 test cases: happy path, missing PRD, missing architecture, missing package.json, concurrent calls (no race)

6. **AC6 — Build gate**
   - `pnpm typecheck` / `pnpm test` / `pnpm lint` all pass, zero regressions.

## Dev Notes

### Why filesystem, not DB

cop1 keeps all state in files (ADR-001). Reading PRD / architecture from their canonical paths guarantees the supervisor reads the same documents the user edits.

### Resolving `planning_artifacts` path

`_bmad/bmm/config.yaml` already exposes `planning_artifacts`. Reuse the existing config loader if one exists; otherwise add a tiny YAML read. Do not hardcode the path — it can be relocated.

### Size ceiling

PRD + architecture can be tens of thousands of tokens. The loader does not truncate; callers (EA10) decide trimming. Note this explicitly in JSDoc to prevent downstream surprises.

### Feature placement — co-locate with `SupervisorService`

`SupervisorContextLoader` lives next to `SupervisorService` in `features/bmad-orchestration/` rather than a separate `features/supervisor/` module. Rationale: all supervisor-scope code stays in one place until a clear reason emerges to split. EA10 orchestrator feature (new `packages/app/src/features/orchestrator/`) is where orchestrator-specific code goes — not a new `sprint-core/features/supervisor/`.
