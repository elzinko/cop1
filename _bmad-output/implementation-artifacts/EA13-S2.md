# Story EA13.2: Real `BMADCommandRunner` default wiring

Status: in-review

## Story

As a **user running `cop1 orchestrator run --epic <id>`**,
I want the CLI to invoke real BMAD commands (not a stub that returns canned transitions),
so that the orchestrator actually drives a sprint ‚Äî calling `/bmad-bmm-create-story`, `/bmad-bmm-dev-story`, `/bmad-bmm-code-review` through a live `BMADSessionPort` ‚Äî and produces real session output, real decisions, and real status transitions.

## Acceptance Criteria

1. **AC1 ‚Äî Default runner is real, not stub**
   - `packages/app/src/cli/commands/orchestrator.ts` no longer defines an inline `stubRunner`. Its default runner invokes `BMADSessionPort.startSession` + `continueSession` for each `(story, command)` tuple.
   - The default wiring follows the pattern from `sprint-run.ts`: `AgentSdkSupervisorAdapter` ‚Üí `SupervisorService` ‚Üí `questionHandler` ‚Üí `AgentSdkSessionAdapter` (or `ClaudeResumeSessionAdapter` if `COP1_BMAD_ADAPTER=resume`).

2. **AC2 ‚Äî Stub runner accessible for tests and dev**
   - Stub moved to `packages/app/src/features/orchestrator/infrastructure/testing/StubBMADCommandRunner.ts`.
   - Exported via a barrel suitable for test imports.
   - A new CLI flag `--runner <default|stub>` (hidden / dev-only) lets operators explicitly fall back to the stub without editing code.

3. **AC3 ‚Äî Contract preserved**
   - The runner returns `{ success, nextStatus?, escalated?, note? }` ‚Äî same shape as the stub, same contract with `OrchestratorService`.
   - `nextStatus` derivation by command matches the stub's existing mapping (`create-story ‚Üí ready-for-dev`, `dev-story ‚Üí in-review`, otherwise ‚Üí `done`).
   - On session failure or error, returns `{ success: false, escalated: true, note: <message> }`.

4. **AC4 ‚Äî Unit test with in-memory adapter**
   - A new test `DefaultBMADCommandRunner.test.ts` verifies:
     - Happy path ‚Äî session completes on first turn; `nextStatus` follows command mapping.
     - Follow-up path ‚Äî multi-turn session drains to completion.
     - Error path ‚Äî session errors produce `{ success: false, escalated: true }`.
     - Timeout path ‚Äî session never completes ‚Üí `{ success: false, escalated: true, note: 'session did not complete' }`.

5. **AC5 ‚Äî Build + existing tests still green**
   - `pnpm build` exit 0.
   - The 9 pre-existing SprintRunner / bmad-pipeline-e2e failures carry over from S1 deferred log; S2 aims to unblock them if natural, else re-defer.

## Dev Notes

### Why not reuse `BMADSessionStep` directly?

`BMADSessionStep` implements `WorkflowStep.run(context: WorkflowContext)`. The orchestrator's runner contract is lighter ‚Äî `{command, storyKey, epicId, projectRoot} ‚Üí {success, nextStatus?, escalated?}`. Forcing a `WorkflowContext` wrapper for every call is boilerplate. Replicate the session lifecycle (start / bounded follow-ups / outcome) inline and document the duplication for an eventual sprint-core extraction.

### Retry / budget concerns

`BMADSessionStep` handles retry + budget via `RetryPolicy` / `BudgetChecker`. For S2 scope, the default runner skips both ‚Äî the orchestrator is the retry boundary (not implemented yet), and budget enforcement is a V1.1 hardening story (EA12-S5 landed single-cap model). Document as "single attempt, no budget gate" in the runner's JSDoc.

### --runner flag visibility

`--runner stub` is a dev/test escape hatch. Mark it "hidden" in commander to avoid surfacing in `cop1 orchestrator run --help`. Operators learn about it from docs/session logs when debugging.

### Pre-existing tests that may naturally re-green

The 9 pre-existing failures in `SprintRunner.test.ts` + `bmad-pipeline-e2e.test.ts` likely stem from the post-EA11-S2 legacy path deprecation. S2 does NOT target them directly ‚Äî but if the real runner wiring produces sensible session defaults, `bmad-pipeline-e2e.test.ts` (which exercises the session pipeline) may unblock. If they remain red, carry forward in `deferred-EA13.md`.
