# Story EA11.4: ADR-013 — Orchestrator vs SprintRunner separation

Status: done

## Story

As a **cop1 architect**,
I want a formal ADR (ADR-013) documenting the separation of responsibilities between `OrchestratorService` (inter-command) and `SprintRunner` (intra-command),
so that EA10 implementation stories and future contributors have an unambiguous reference for which layer owns which concern, and so that the boundary survives future refactors.

## Acceptance Criteria

1. **AC1 — ADR file produced**
   - File `_bmad-output/planning-artifacts/adr-013-orchestrator-sprintrunner-separation.md` exists and follows the ADR template used elsewhere in the folder (title, status, context, decision, consequences, alternatives considered).
   - Status field: `Accepted`.

2. **AC2 — Separation clearly stated**
   - The ADR explicitly assigns: **intra-command** concerns (worktree setup, session lifecycle, checkpoint, story filtering, event emission) to `SprintRunner`.
   - **Inter-command** concerns (playbook reading, command sequencing, multi-step supervisor loop, auto-decision logging) to `OrchestratorService`.
   - A diagram or table shows the request flow: `CLI → OrchestratorService → SprintRunner → BMADSessionPort → <command>`.

3. **AC3 — Rationale covers scope creep risks**
   - ADR discusses why these are two services, not one: testability, playbook evolution, supervisor invocation surface.
   - ADR lists concerns explicitly NOT in scope for `OrchestratorService`: intra-command step control (remains in `SprintRunner`), tool exposition (covered by ADR-014).

4. **AC4 — Cross-links**
   - ADR references ADR-012 (BMAD session port), ADR-014 (supervisor tool interface, if already committed; placeholder otherwise), and the epic EA10 user value.
   - epics.md §EA10 and §EA11 reference ADR-013 where appropriate (already done in some places — verify consistency).

5. **AC5 — Review & commit**
   - ADR reviewed by user (architect session) before commit.
   - Commit message includes `ADR-013` and references EA11-S4.
