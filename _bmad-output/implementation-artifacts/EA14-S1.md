# Story EA14-S1: BMAD bundle detection pre-flight guard

Status: done

## Story

As a **developer running `cop1 orchestrator run`**,
I want the orchestrator to fail fast with a clear error when `_bmad/` is absent at `projectRoot`,
so that I don't get a silent fiction run that reports success without any real BMAD execution.

## Acceptance Criteria

1. **AC1 — Pre-flight check for `_bmad/` directory**
   - Before building the default runner, check that `path.join(projectRoot, '_bmad')` exists (directory or symlink).
   - If absent, error with message: "No BMAD installation found at <path>/_bmad. The orchestrator requires BMAD workflows to execute real sprint commands. Install BMAD or use --runner stub (with COP1_ALLOW_STUB_RUNNER=1) for testing."

2. **AC2 — Stub runner bypasses the check**
   - The check is skipped when `--runner stub` is used (stub doesn't need BMAD).

3. **AC3 — Unit tests cover both branches**
   - Directory present -> proceed.
   - Directory absent -> exit code 2 (runtime error via thrown Error caught by CLI).

4. **AC4 — Existing tests still pass**
   - `orchestrator.test.ts` and `orchestrator-real-run.test.ts` tests pass. Those tests use `overrides.runner` which bypasses the default runner construction.

5. **AC5 — `pnpm build` green**

## Dev Notes

### Implementation choice

The guard is placed in `resolveRunner()` after the stub runner early-return. This means:
- `--runner stub` bypasses the check entirely (AC2).
- `overrides.runner` in tests bypasses `resolveRunner()` entirely, so the guard never fires for test harnesses (AC4).
- The guard throws an `Error`, which is caught by the existing `try/catch` in `orchestratorRunCommand` and mapped to exit code 2. This is consistent with the "runtime error" exit code pattern already used.

### Why `existsSync` and not `stat`

The check is a synchronous pre-flight guard at startup. There is no performance concern (single syscall, runs once before the entire sprint loop). `existsSync` is simpler and avoids an unnecessary async boundary.
