# Story EA10.7: Multi-agent advisory capability for Supervisor

Status: done

## Story

As a **cop1 Supervisor faced with a blocked decision**,
I want a concrete implementation of the ADR-014 tool interface (MCP / Agent SDK tools / sidecar — per ADR-014) that lets me consult external agents and cop1 technical services (worktree, checkpoint, history, advisory agents) to gather context and resolve the block,
so that the multi-step resolution loop (EA10-S8) has real capabilities behind its "consult agents" phase, instead of a stub.

## Acceptance Criteria

1. **AC1 — In-process MCP server via `createSdkMcpServer`** (ADR-014 §4.2)
   - Supervisor session wires an **in-process MCP server** built with `createSdkMcpServer` + `tool` helpers from `@anthropic-ai/claude-agent-sdk`
   - The MCP server is registered via the `query()` options when the supervisor session starts (per Agent SDK MCP API)
   - The MCP server is named `cop1-supervisor-tools` (ADR-014 §4.2 code sample)
   - No standalone process, no stdio IPC — purely in-process

2. **AC2 — 3-layer architecture** (ADR-014 §4.3)
   - **Layer 1 (core logic):** Pure business functions (e.g., `createWorktreeCore(input): Output`) live in the technical services extracted by EA11-S3 (`WorktreeService`, `HistoryService`, etc.). No SDK dependency in this layer.
   - **Layer 2a (SDK wrapper):** Each tool wraps Layer 1 via `tool(name, description, zodSchema, handler)`. Wrapper handles input validation, output shaping, and EventBus emission. ~30 lines of boilerplate per tool (explicitly accepted in ADR-014 §4.3).
   - **Layer 2b (future standalone MCP):** NOT built in V1-light. Layer 1 stays free of SDK imports so a future standalone MCP wrapper can reuse it without refactor.

3. **AC3 — Tool catalog V1-light** (ADR-014 §4.4)
   - Required tools, each with Zod schema + Layer 1 binding:
     - `create_worktree` → `WorktreeService.create`
     - `cleanup_worktree` → `WorktreeService.cleanup`
     - `invoke_bmad_command` → `BMADSessionPort.startSession` (with re-entrance depth counter, cap 3 — ADR-014 §5.7)
     - `query_session_history` → `HistoryService.byStory` / `bySession` (naming per ADR-014 §4.4 — NOT `recall_narrative_log`)
     - `read_checkpoint` → `CheckpointService.read`
     - `save_checkpoint` → `CheckpointService.save`
     - `commit_anchor` → git commit one workflow at a time (ADR-014 §5.6)
     - `remaining_budget` → returns token budget remaining for the long-running session

4. **AC4 — Task 0 — Agent SDK MCP spike**
   - Before implementation: 30-minute spike against the real SDK to validate:
     - `createSdkMcpServer` API shape + wiring into `query()` options
     - Tool invocation flow (tool call → handler → response back to LLM)
     - Error surfaces when a tool handler throws
     - Scope isolation (confirm BMAD internal agents do NOT see these tools — ADR-014 §5.1/§5.2)
   - Findings documented in `_bmad-output/planning-artifacts/ea10-s7-spike.md`; reviewed before implementation task starts

5. **AC5 — Supervisor-only scope** (ADR-014 §5.1–§5.3)
   - MCP server attached **only** to the supervisor session. BMAD internal agent sessions (invoked via `invoke_bmad_command`) get a pristine environment — no environment variable, no config, no awareness of cop1 tools.
   - Single point of contact: cop1 ↔ BMAD goes exclusively through `invoke_bmad_command`. No out-of-band channel.

6. **AC6 — Integration with `SupervisorService`**
   - `SupervisorService` (EA9-S3) receives the MCP server handle at construction or via a `registerTools()` method
   - Tool invocation events emitted on `EventBus`: `supervisor.tool.invoked`, `supervisor.tool.completed`, `supervisor.tool.failed` — each with tool name, input hash, duration, outcome

7. **AC7 — Unit + integration tests**
   - Unit: each tool's Layer 2a wrapper tested with mocked Layer 1 service
   - Unit: re-entrance depth counter on `invoke_bmad_command` (cap at 3, returns error beyond)
   - Integration: `SupervisorService` + MCP server + at least 2 tools exercised end-to-end in a scripted scenario via `InMemorySessionAdapter`

8. **AC8 — Build gate**
   - `pnpm typecheck` / `pnpm test` / `pnpm lint` pass, zero regressions

## Dev Notes

### Concrete mechanism (from ADR-014)

The abstract "per ADR-014" has resolved: **in-process MCP via `createSdkMcpServer`**. All ACs now reflect that concrete decision. Code sample from ADR-014 §4.2:

```typescript
import { tool, createSdkMcpServer } from "@anthropic-ai/claude-agent-sdk";
import { z } from "zod";

const createWorktree = tool(
  "create_worktree",
  "Create a git worktree for a story and return its path",
  { storyId: z.string(), projectPath: z.string() },
  async ({ storyId, projectPath }) => {
    return worktreeServiceCore.create(storyId, projectPath);
  }
);

const mcpServer = createSdkMcpServer({
  name: "cop1-supervisor-tools",
  tools: [createWorktree, invokeBmadCommand, querySessionHistory, /* ... */]
});

// When starting supervisor session:
query({ prompt, options: { mcpServers: { 'cop1': mcpServer }, /* ... */ } });
```

### Risk #1 & #4 mitigations (epics.md §EA10)

- **Risk #1 (ADR-014 quality):** ADR-014 committed (`539db6b`). Decisions translated into ACs here. Task 0 spike provides the final validation.
- **Risk #4 (Agent SDK integration fragility):** Task 0 spike catches surprises before larger commit. EA9 retro lesson: real SDK > assumed SDK.

### Why 3 layers — don't skip Layer 1 purity

ADR-014 §4.3 explicitly calls out ~30 lines of boilerplate per tool as the price of long-term portability. Accept it. When V1.1+ introduces a second consumer (e.g., standalone MCP for iamthelaw agent per §5.5), the Layer 1 core reuses without SDK coupling. Short-cutting by embedding SDK calls in core functions breaks the contract.

### Tool granularity

Keep each tool atomic (one verb, one input schema, one output schema). Coarse tools invite misuse. Resist composite tools. ADR-014 §4.4 lists the V1-light catalog — do not add tools outside this list without an ADR amendment.
