# Story EA10.6: CLI `cop1 orchestrator run --epic <id>`

Status: done

## Story

As a **cop1 Developer**,
I want a `cop1 orchestrator run` CLI subcommand with flags `--playbook`, `--epic`, `--step-by-step`, `--abort-on-escalation`,
so that I can trigger the Supervisor Orchestrator from the terminal without writing a bespoke script, and so that the CLI wiring of `OrchestratorService` to its dependencies lives in a single composition point.

## Acceptance Criteria

1. **AC1 — Command registered**
   - `cop1 orchestrator run` registered in the CLI
   - Discoverable via `cop1 --help` and `cop1 orchestrator --help`

2. **AC2 — Flags**
   - `--playbook <path>` (default: `supervisor-playbook.md` at project root)
   - `--epic <id>` (required)
   - `--step-by-step` (boolean, default false)
   - `--abort-on-escalation` (boolean, default false)
   - `--project-root <path>` (optional, default: `process.cwd()`)

3. **AC3 — Composition root**
   - New composition entry `packages/app/src/composition/OrchestratorRunner.ts` (or similar) wires:
     - `SupervisorPlaybookLoader` (EA10-S1)
     - `OrchestratorService` (EA10-S4)
     - `SupervisorService` + `SupervisorContextLoader` + `SprintRunner`
     - `WorktreeService` + `HistoryService` + `StepByStepController`
     - `EventBus`
     - JSONL logger to `.cop1/sprint-log-*.jsonl`

4. **AC4 — Exit codes**
   - 0 success (all stories processed)
   - 1 invalid arguments / missing playbook / unknown epic
   - 2 runtime error during execution
   - 3 escalation under `--abort-on-escalation`

5. **AC5 — Help text**
   - Examples: running on an epic, running step-by-step, running in CI with file-based approval

6. **AC6 — Integration test**
   - Test invoking the CLI against a fixture project (mocked `BMADSessionPort`), asserts end-to-end flow + exit codes

7. **AC7 — Build gate**
