# Story EA10.4: `OrchestratorService` main loop (1-epic scope)

Status: done

## Story

As a **cop1 Developer kicking off `cop1 orchestrator run --epic <id>`**,
I want an `OrchestratorService` that reads the playbook, iterates over each story of the target epic, and drives the canonical BMAD command sequence (create-story → dev-story → code-review) by delegating each command to `SprintRunner` via `BMADSessionPort` per ADR-013,
so that a single epic gets executed end-to-end with state persisted to `sprint-status.yaml` on every transition and zero human intervention in the happy path.

## Acceptance Criteria

1. **AC1 — `OrchestratorService` exists**
   - Class `OrchestratorService` in `packages/app/src/features/orchestrator/application/OrchestratorService.ts`
   - Constructor injects: `SupervisorPlaybookLoader`, `SprintRunner`, `SupervisorService` (EA9-S3), `SupervisorContextLoader` (EA11-S6), `WorktreeService` + `HistoryService` + `StepByStepController` (EA11-S3), `EventBus`
   - Public `run(options: { playbookPath: string, epicId: string, mode: OrchestratorMode }): Promise<OrchestratorRunResult>`

2. **AC2 — Epic scope iteration**
   - Reads `sprint-status.yaml` to list stories belonging to `epicId`
   - Stops on first story that is not in status `ready-for-dev` or earlier state (already-done stories skipped)
   - Processes stories in file order

3. **AC3 — Canonical command sequence per story**
   - For each story, executes the playbook phases in order:
     1. `create-story` (if story file does not exist yet)
     2. `dev-story`
     3. `code-review`
   - Each command invoked through `BMADSessionPort.startSession` (via `SprintRunner` per ADR-013)

4. **AC4 — State persistence on every transition**
   - After each command completes, update the corresponding entry in `sprint-status.yaml`:
     - After `create-story` success → `ready-for-dev`
     - After `dev-story` success → `in-review`
     - After `code-review` success → `done`
     - On any failure → `blocked`
   - Writes preserve YAML comments and structure (use existing `sprint-status` writer helpers or shell out to the sprint-core YAML util)

5. **AC5 — Abort on supervisor escalation**
   - When `SupervisorService` exhausts its multi-step loop (EA10-S8) and escalates, the orchestrator:
     - If mode is `abort-on-escalation` → stops cleanly, persists current state, emits `orchestrator.escalated` event, exits with non-zero code
     - Otherwise → logs escalation and continues with the next story (configurable)

6. **AC6 — Auto-decision logging**
   - Every supervisor decision (deterministic, LLM-inferred, consult, synthesized) is logged as structured event level `auto-decision` in `.cop1/sprint-log-{date}.jsonl`
   - Payload includes: `decisionType`, `input`, `output`, `llmJustification?`, `storyId`, `command`, `ts`

7. **AC7 — EventBus emissions**
   - `orchestrator.run.started`, `orchestrator.story.started`, `orchestrator.command.started`, `orchestrator.command.completed`, `orchestrator.story.completed`, `orchestrator.run.completed` — all with timestamps + ids

8. **AC8 — Unit tests**
   - Test file `packages/app/src/features/orchestrator/__tests__/OrchestratorService.test.ts`
   - Mock `BMADSessionPort` (inject scripted sessions), mock `SupervisorService`, fixture `sprint-status.yaml`
   - Cover: happy path single story, happy path multi-story, failure mid-command, escalation path, state-machine transitions

9. **AC9 — Build gate**
   - `pnpm typecheck` / `pnpm test` / `pnpm lint` pass, zero regressions

## Dev Notes

### ADR-013 boundary

`OrchestratorService` = inter-command. `SprintRunner` = intra-command. Do NOT replicate intra-command logic (worktree setup, session lifecycle, checkpoint) here.

### Dependency chain

Depends on: EA10-S3 (playbook), EA11-S3 (services extracted), EA11-S4 (ADR-013 approved), EA11-S5 (ADR-014 approved), EA11-S6 (SupervisorContextLoader), EA9 (done).

### `sprint-status.yaml` is BMAD-owned

Per ADR-009: BMAD is the source of truth. cop1 reads but writes via the same BMAD-compatible path (YAML preserved as-is). Use the existing writer.

### No advanced policies

V1-light scope: 3-tier cascade for supervisor, no plan validation / elicitation selection / response validation. Those are V1.1 (original EA10-S3 deferred).
